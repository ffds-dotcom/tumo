▼

10:15 – 11:00

Course Overview + Live Demo + Prompt Engineering

▼

Part A — Prompting Rules

  1. Break the chess game down into individual rules and implement them one at a time — never ask Claude to implement multiple things in a single prompt
  2. Make small, focused commits — each commit should represent one working change
  3. After every change, test and validate that Claude did it correctly before moving on — once confirmed, commit it so it stays in history
  4. If you don't understand what Claude did, ask it to explain — don't move on until you understand the change
  5. Even better: ask Claude to generate a webpage that explains all the changes it made — so you always have a readable record of what was built and why
Work in baby steps. Small changes are critical because:
  1. You will understand what the AI does better if it goes step by step
  2. You can correct the AI faster
  3. You can catch and describe errors more easily

Part B — Course Overview (10 min)

  • Walk through the 10-day arc on a whiteboard or slide
  • Show the endgame: a chess app with bot opponent
Say aloud: "Over 10 days you'll build a fully playable chess game using AI as your coding partner. You'll learn to give AI precise instructions, use Git so your work is never lost, write unit tests so you can trust your code, and ship something real."

Part C — Live Demo: Build a Game in Front of the Class (20 min)

  • Open claude.ai, share screen
  • Start with a vague prompt ("make me a game") → show weak output
  • Re-prompt precisely → show the improvement → compare to platformer.html as the polished version
  • Point out a bug live → "this is what we fix as developers"
Key message: AI is fast but it needs a skilled human steering it. You are that human.

Part D — Claude Models + Prompt Engineering Principles (10 min)

  • Haiku — fast, cheap, good for simple tasks
  • Sonnet — balanced, your daily driver
  • Opus — most powerful, use for hard problems
  1. Be specific — give context, describe the goal, not just the symptom
  2. Describe expected vs. actual behavior when debugging
  3. Iterate — bad output is feedback, not failure
  4. Always read before you paste — AI can introduce bugs silently

Exercise — Paper Prompt (5 min)

  • Ask: "Write the prompt you'd give Claude to build a game where a ball bounces around the screen."
  • Students write on paper — no computers
  • Compare 3–4 answers aloud: who was vague? who was specific?
  • Use their answers to segue directly into the 4 principles above

Day 3 — June 24

Chess Board UI
9:30 – 11:30

Build the Chess Board

▼
  • Students choose their language/tech stack (default: HTML/CSS/JS)
  • Create a new project workspace + Git repo: chess-app
  • First commit: empty project scaffold
  • Task: build a chess board — 8×8 grid, alternating colors, correct dimensions
11:30 – 1:30

Teacher Review

▼
  • Teacher reviews each group's board against UI requirements
  • Review code and Git history — look for meaningful commit messages
Hard rule: Students cannot move on to pieces until the board is fully complete and approved.

Board UI Requirements (minimum)

  • 8×8 grid with correct alternating light/dark squares
  • Board is visually centered on the page
  • Squares are equal size and proportional
  • Board has file (a–h) and rank (1–8) labels

Day 5 — June 26

Fully Functional Two-Player Chess
9:30 – 1:30

Continuation of Day 4

▼

Goals for This Day

  • Players can take turns (white moves, then black, then white...)
  • Capturing works (moving onto an opponent's piece removes it)
  • Basic illegal move blocking (can't move to a square occupied by your own piece)
  • Win detection: game ends when a king is captured (or checkmated — stretch goal)
Teacher role: Work with every group. No group should leave Day 5 without a game where two people can sit down and play.

Stretch Goals

  • Check detection and visual indicator
  • Highlight valid moves on selection
  • Turn indicator UI
End of Day Goal Two people can sit down and play a real game of chess, even if rule enforcement is incomplete.

Day 6 — June 29

Rule Enforcement · Unit Testing
9:30 – 11:30

Full Rule Enforcement

▼
  • Each piece type follows its legal movement rules
  • Illegal moves are blocked (not just same-color squares)
  • Special moves: castling, en passant, pawn promotion
  • UI shows highlighted valid moves when a piece is selected
  • Students prompt Claude to implement and test each rule one at a time
11:30 – 1:30

Unit Testing

▼
  • Teacher lecture: what unit tests are and why they exist
  • Live demo: show how a unit test catches a bug that AI introduced silently
  • Students write unit tests for their move validation logic
  • Unit tests must be integrated into prompts going forward: "write a function and a unit test for it"
End of Day Goal Game enforces legal moves for all piece types; at least 5 unit tests written and passing.

Day 7 — June 30

Chess Bot — Most Challenging Day
9:30 – 11:30

Bot Algorithm Lecture + Planning

▼
  • Random move bot (baseline)
  • Minimax algorithm (look ahead N moves)
  • Basic evaluation function — material value: pawn=1, knight/bishop=3, rook=5, queen=9
  • Students plan their bot: algorithm choice, depth, how to test it
Rule: No third-party chess libraries. Algorithm must be self-implemented with AI help.
11:30 – 1:30

Bot Build + Validation

▼
  • Students build the bot using Claude
  • Teacher works with each group on the algorithm
  • Students write tests to validate bot behavior (does it avoid losing its queen? does it take free pieces?)
End of Day Goal A bot that can play a legal game of chess against the player, even if it plays poorly.

Day 8 — July 1

Custom Chess Variant · Precision Prompting Challenge
9:30 – 1:30

Variant Challenge

▼

Each group receives a unique challenge designed to force precise prompt engineering. Vague prompts won't work — the challenge is too specific.

4-Player ChessTwo extra sets of pieces, 4-corner board
Never-Ending ChessCaptured pieces respawn after N turns
Queens OnlyAll pieces replaced with queens
Fog of WarOnly see squares your pieces can move to
Bot vs. BotTwo bots fighting, no human player
Speed ChessEach player has 30 seconds per move
Blindfold ModeNo pieces shown, only move notation
Teacher role: Assign variants based on group ability — harder variants for stronger groups.
Goal: Students must describe the variant precisely to Claude in order to implement it without wasting tokens on bad output. Prompt quality is the lesson.
